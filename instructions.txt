   ###################### 
   ВЫПУСК СЕРТИФКАТА НА GITLAB

   Нужно импортировать сертификат GitLab в Java, которая установлена на сервере SonarQube,
   иначе будет ошибка в Sonar "Could not validate GitLab url. Got an unexpected answer"

   Перед выпуском сертификата, сперва нужно куда-нибудь переместить тот сертификат 
   (открытая и закрытая его части), который был создан при установке GitLab 
   (он подходит для ГитКонвертер, но не подойдёт потом для SonarQube). 
   По умолчанию сертификаты в GitLab находятся в /etc/gitlab/ssl/

   Для выпуска сертификата при помощи OpenSSL нужно выполнить следующую команду:

   sudo openssl req -x509 -nodes -days 1825 -newkey rsa:2048 \
    -subj "/C=RU/ST=MO/L=MO/O=EXP/OU=EXP/CN=<gitlab-hostname>" \
    -addext "subjectAltName = DNS:<gitlab-hostname>, IP:<gitlab-hostip>" \
    -keyout <path-to-cert>/<gitlab-hostname>.key \
    -out <path-to-cert>/<gitlab-hostname>.crt

   , где:
    * <gitlab-hostname> - имя хоста gitlab, напр. gitlab.example.com
    * <gitlab-hostip> - ip хоста, на котором стоит gitlab (ip докер контейнера)
    * <path-to-cert> - путь, куда класть сертификаты (можно не указывать - в текущий каталог)
    * <user> - имя пользователя, в чей каталог будут сброшены сертификаты 

    обязательно поменять права на файлы chmod 644 <path-to-cert>/<gitlab-hostname>*

    Сгенерированные файлы сертификата (.crt, .key) помещаем в каталог /etc/gitlab/ssl/

    !!!ВНИМАНИЕ!!! на latest версии gitlab-EE ошибка чтения параметра letcencrypt
    ----
      В файле /etc/gitlab/gitlab.rb отключаем Let’s Encrypt, чтобы GitLab сам себе не выпустил сертификат и не затёр тот, который сгенерирован при помощи OpenSSL:
      letcencrypt['enable'] = false

      После этого нужно выполнить команду: sudo gitlab-ctl reconfigure. Дожидаемся окончания выполнения команды.
    ----

    НАСТРОЙКА SONARQUBE

    1. Экспортируем сертификат с GitLab
    openssl s_client -connect <gitlab-hostname>:443 -showcerts </dev/null 2>/dev/null | openssl x509 -outform PEM > gitlab-cert.pem

    2. Или используем существующий
    cp /home/nikita/cert-gitlab-openssl/<gitlab-hostname>.crt gitlab-cert.pem
 
    3. Копируем сертификат в контейнер
    docker cp gitlab-cert.pem sonarqube-container:/tmp/gitlab-cert.pem

    4. Импортируем в Java truststore
    docker exec --user=root sonarqube-container keytool -importcert -keystore /opt/java/openjdk/lib/security/cacerts -storepass changeit -alias gitlab -file /tmp/gitlab-cert.pem -noprompt

    5. Перезапускаем SonarQube
    docker restart sonarqube-container



    ######################
    НАСТРОЙКА GITLAB-RUNNER (DOCKER)

    1. готовим каталоги:

    sudo mkdir -p /srv/gitlab-runner/{config,logs,certs}
    sudo chmod -R 755 /srv/gitlab-runner

    2. кладем сертификаты в раннер:
    openssl s_client -connect <gitlab-hostname>:443 -showcerts </dev/null 2>/dev/null | openssl x509 -outform PEM > gitlab.crt
     *  <gitlab-hostname> = gitlab.example.com 

    sudo cp /path/to/your/gitlab/certificate.crt /srv/gitlab-runner/certs/gitlab.crt

    3. Запускаем контейнер

    4. добавляем runner в gitlab, копируем токен для нового раннера

    5. Регистрируем runner
    docker exec -it gitlab-runner bash

    # Регистрируем runner с указанием сертификата
    gitlab-runner register  --url <gitlab-hostname>  --token <runner-token> --tls-ca-file "/etc/gitlab-runner/certs/gitlab.crt"
    указываем имя раннера (напр. docker-runner-1)
    выбираем executor (напр. docker или shell)
    если docker, выбираем базовый образ (напр. alpine:latest)
    
    6. Проверяем в gitlab наличие раннера

    7. Меняем конфиги раннера:
    [runners.docker]
      # Добавляем
      volumes = ["/cache", "/var/run/docker.sock:/var/run/docker.sock"]
      extra_hosts = ["<gitlab-hostname>:<ip-host>"] # <ip-host> - хост на котором стоит докер контейнер 
      dns = ["8.8.8.8", "8.8.4.4"]

    2. кладем сертификаты в этот же каталог раннера:

    указываем имя раннера (напр. windows-runner-1)
    выбираем executor (напр. shell:pwsh)
    
    6. Проверяем в gitlab наличие раннера

2. ИНТЕГРАЦИЯ GITLAB И SONARQUBE
---------------------------------------------

    1. Создаем отдельного пользователя в GitLab:
      name: sonarqube
      pass: {password}

      делаем ему через его учетную запись токен авторизации (можно сунуть все права чтобы не было проблем),
      токен сохраняем - он будет нужен для sonarqube

      Пользователя включаем в нужные проекты, чтобы был доступ до проектов.

    2. Создаем отдельного пользователя в SonarQube:
      name: gitlab
      pass: {password}

      делаем токен авторизации - он будет нужен для gitlab

    3. Коннектим gitlab в sonarqube 

       Администрирование - Интеграция с ALM - GitLab, там надо будет указывать хост gitlab и токен из п.1

       Конфигурация должна чекаться иначе идти в логи контейнера и разбираться, возможно не резолвится хост либо не прокинут сертификат в SonarQube

    4. Импортируем проект в sonarqube (Проекты - Создать проект - Из gitlab)

    5. Заходим в проект - настройки проекта - Разрешения - чекаем флаги для юзера gitlab

    6. В gitlab в настройках проекта - CI/CD - задаем переменные: 
        
       SONAR_HOST_URL - URL хоста SonarQube (http://<ip-host>:9000)
       SONAR_TOKEN - токен из п. 2

    7. подкидываем .gitlab-ci.yml в корень проекта - проверяем pipeline по PR в ветку (текущая конфа на ветку develop), разбираем ошибки

    8. чтобы работали декораторы PR из Sonar следует обязательно проверить настройки:
      - Настройки проекта - Интеграция с платформой DevOps 
      - Администрирование - Общие - Images base URL / Server base URL
      -

НАСТРОЙКА ВЕБ СЕРВЕРА

Версия для apache 2.4.

В файле apache2.conf добавляем:

    LoadModule _1cws_module /opt/1cv8/x86_64/<ВерсияПлатформы>/wsap24.so

    Alias "/repos" "/var/1C/www/repos"
    <Directory "/var/1C/www/repos">
        AllowOverride None
        Options None
        Require all granted    
        SetHandler 1c-application
    </Directory>

где
    /repos - адрес ресурса (напр. для каталога с храном test1 будет путь .../repos/test1)
    /var/1C/www/repos - путь до каталога файла *.1ccr
    
    В папке корневого каталога хранилищ создать файл *.1ccr, напр.: repo.1ccr:
    <?xml version="1.0" encoding="UTF-8"?>
    <repository connectString="tcp://127.0.0.1"/>

    где tcp://127.0.0.1 - адрес сервера хранилищ

ВАЖНО: не путать каталоги: 
    ПутьККаталогуХранилищ - каталог, где физически будут созданы хранилища - нужны права пользователю USR1CV8
    /var/1C/www/repos - путь к файлу *.1ccr - нужны права пользователя apache2 (в linux - www-data)

Доступ к хранилищу осуществляется на примере выше:
    tcp://localhost/test1
    http://localhost/repos/repo.1ccr/test1

Для linux создаем службу: /etc/systemd/system/crs1cv8-8.3.27.2074.service

    [Unit]
    Description=1C:Enterprise CRS Server 8.3 (8.3.27.2074)
    [Service]
    ExecStart=/opt/1cv8/x86_64/8.3.27.2074/crserver -d /home/usr1cv8/.1cv8/1C/1cv8/repos
    RemainAfterExit=yes
    User=usr1cv8
    Group=grp1cv8
    [Install]
    WantedBy=multi-user.target

в этом случае каталог хранилищ - /home/usr1cv8/.1cv8/1C/1cv8/repos

НАСТРОЙКА GIT SYNC
---------------------------------------------------------------

устанавливаем gitsync
opm install gitsync

ставим плагины sync-remote, increment

    gitsync p init
    gitsync enable sync-remote
    gitsync enable increment

создаем каталог для локального git репозитория
напр. E:/GIT/test/main + E:/GIT/test/bit_ДоработкиАдаптация

создаем на git сервере репозиторий
в каталоге git (E:/GIT/test) инициализируем репозиторий
git init
подключаем origin по инструкции git сервера, типа:
    cd existing_repo
    git remote add origin https://gitlab.userv.ru/firstbit/testrepos.git
    git branch -M main
    git push -uf origin main

добавлем ConfigDump  в .gitignore

gitsync init --storage-user ServiceUser --storage-pwd 123654 tcp://localhost:2555/test/main E:\GIT\test\main
gitsync init --storage-user ServiceUser --storage-pwd 123654 -e bit_ДоработкиАдаптация tcp://localhost:2555/test/bit_ДоработкиАдаптация E:\GIT\test\bit_ДоработкиАдаптация

В каталогах появляются файлы AUTHORS и VERSION
Правим файлы AUTHORS, привязывая пользователей:
Администратор=Администратор <Администратор@localhost>
IIIvanov=Ivan Ivanov <iiivivanov@test.ru>

выполняем первичную синхронизацию:
    gitsync sync -u ServiceUser -p 123654 -d --PS -b develop -i tcp://localhost:2555/test/main E:\GIT\test\main https://gitlab.userv.ru/firstbit/testrepos.git
      
    gitsync sync -u ServiceUser -p 123654 -e bit_ДоработкиАдаптация -d --PS -b develop -i tcp://localhost:2555/test/bit_ДоработкиАдаптация E:\GIT\test\bit_ДоработкиАдаптация https://gitlab.userv.ru/firstbit/testrepos.git

По умолчанию gitsync использует временную файловую БД для выполнения операций с хранилищем, это занимает некоторое время для ERP.
Можно подкинуть параметр --ibconnection с путем к базе, подключенной к хранилищу (до команды sync{s})

gitsync  
    -C /Slocalhost/test_dev_2 // путь к базе, для аутентификации (если нужна) добавить параметры -U user -P password
    s                         // команда sync
    -u ServiceUser -p 123654  // аутентификация в хранилище
    -d                        // не искать src
    --PS                      // [sync-remote] плагин - отправляем изменения в origin 
    -b develop                // [sync-remote] плагин - имя ветки
    -t postgresql             // [*use-ibcmd] плагин - тип СУБД 
    -n test_dev_2             // [*use-ibcmd] плагин - имя БД 
    -U ServiceUser -P 123654  // [*use-ibcmd] плагин - аутентификация в БД
    -j 4                      // [*use-ibcmd] плагин - Количество потоков при экспорте 
    -i                        // [*use-ibcmd] плагин - инкрементальная выгрузка конфигурации
    tcp://localhost:2555/test/main                  // путь к хранилищу
    E:\GIT\test\main                                // путь к каталогу в git репозитории
    https://gitlab.userv.ru/firstbit/testrepos.git  // адрес удаленного репозитория

gitsync -C /Slocalhost/test_dev_2 -U ServiceUser -P 123654 s -e bit_ДоработкиАдаптация -u ServiceUser -p 123654 -d --PS -b develop -t postgresql -n test_dev_2 -U postgres -P 5g2bMD5sMPG3 -j 4 -i tcp://localhost:2555/test/bit_ДоработкиАдаптация E:\GIT\test\bit_ДоработкиАдаптация https://gitlab.userv.ru/firstbit/testrepos.git

ВНИМАНИЕ!!! ПРОБЛЕМЫ В РАБОТЕ С РАСШИРЕНИЯМИ:

    - расширения служебной БД должны быть отключены от хранилищ (падает в ошибку)

    gitsync -C /Slocalhost/test_dev_2 -U ServiceUser -P 123654 s -u ServiceUser -p 123654 -d --PS -b develop tcp://localhost:2555/test/main

OPROXY: СОЗДАНИЕ ПАЙПЛАЙНА GITLAB ДЛЯ СОЗДАНИЯ ОЧЕРЕДЕЙ ЗАДАНИЙ НА ЗАПУСК GITSYNC
-----------------------------------------------------------------------------------
Чтобы избежать одновременного запуска oscript с использованием одного и того же репозитория (при одновременном помещении в хранилище несколькими пользователями), сделаем вместо явного запуска gitsync обращение к api pipeline gitlab - будет строиться очередь выполнения - все отработает штатно.

для обрашения по api необходимо получить trigger token в настройках проекта - CI/CD - Pipeline trigger token.

Пример обращения по апи для старта gitlab (из документации):
    curl --request POST \
     "https://gitlab.example.com/api/v4/projects/<project_id>/trigger/pipeline?token=<token>&ref=<ref_name>"

    Код ответа при успешном запуске = 201

ИНТЕГРАЦИЯ SONARQUBE  С GITLAB
-----------------------------------------------------------------------------------------
расширяем pipeline сонаром:
sonarqube-analysis:
  stage: sonarqube-check
  variables:
    SONAR_USER_HOME: "${CI_PROJECT_DIR}/.sonar"
    GIT_DEPTH: "0"                          # Нужно для истории изменений
    GIT_LFS_SKIP_SMUDGE: "1"
    GIT_STRATEGY: fetch
  image: 
    name: sonarsource/sonar-scanner-cli:latest
  script:
    - sonar-scanner -Dsonar.host.url="${SONAR_HOST_URL}"
  allow_failure: false 
  rules:
    # Запускаем при push в develop (не merge request)
    - if: $CI_COMMIT_BRANCH == 'develop' && $CI_PIPELINE_SOURCE == 'push'
      changes:
        - "**/*.bsl"
        - "**/*.os"
  tags:
    - sonar

Сонар начинает сканить все подряд, несмотря на настройку with support lock
